# Three-Person Build Plan: Next.js Frontend, LangGraph PDF Processing, TinyFish Eligibility and Scheduling

## Summary

Build the app as a guided demo with a clear division of responsibilities between the two agent systems:

- **LangGraph** handles the referral packet itself:
  - OCR / document text extraction
  - field extraction
  - cross-field validation
  - final structured patient/referral record
- **TinyFish** handles browser-style operational checks against dummy web pages:
  - service ZIP coverage
  - accepted insurance plans
  - nurse specialization and schedule lookup
  - patient outreach / scheduling action flow

The user journey stays:

`Homepage -> Referral Portal -> Referral Review -> Process PDF -> Extracted Patient Info -> Eligibility Check -> Scheduling/Call Orchestration -> Final Scheduled State`

Platform layout:
- **Frontend**: Next.js + shadcn/ui on Vercel
- **Backend**: Node service on AWS App Runner
- **Database / file storage**: Supabase Postgres + Storage
- **Workflow runtime**: LangGraph for document-processing graph
- **Browser automation/runtime**: TinyFish for eligibility and scheduling workflows

## App Flow in Product Terms

1. User lands on the homepage and clicks `Demo`.
2. User sees the referral portal with a few seeded referrals.
3. Clicking a referral opens a detail page:
   - PDF on the right
   - currently known referral fields on the left
4. Clicking `Process / Extract` starts a **LangGraph document-processing workflow**:
   - OCRs the PDF
   - extracts important fields
   - validates and reconciles them
5. User is taken to a clean extracted-data page showing:
   - patient demographics
   - MRN
   - phone
   - address
   - insurance
   - ordered services
   - physician details
6. From that structured patient record, a **TinyFish eligibility workflow** runs against two dummy pages:
   - agency ZIP coverage page
   - agency accepted insurance page
7. If eligibility passes, a second **TinyFish scheduling workflow** runs:
   - reads extracted patient info
   - reasons over ordered services
   - checks dummy nurse specialization/schedule pages
   - schedules a visit/call
   - simulates patient outreach
8. User sees a final scheduling result:
   - eligible / not eligible
   - assigned nurse
   - proposed visit/call slot
   - scheduling initialized / blocked

## Core Architecture

### Frontend
Use **Next.js App Router** with **shadcn/ui** and TypeScript.

Route mapping:
- `/` homepage
- `/portal`
- `/referral/[id]`
- `/processing/[id]`
- `/result/[id]`
- `/eligibility/[id]`
- `/schedule/[id]`

If preferred, `eligibility` can be folded into `result` or `schedule`, but keeping it separate makes the story clearer and avoids overloading one screen.

### Backend
Use a persistent Node service with:
- `Express`
- `LangGraph.js`
- `@supabase/supabase-js`
- `zod`
- TinyFish SDK
- structured logging
- SSE endpoints

### Supabase
Use Supabase for:
- relational workflow state
- extracted patient data
- seeded nurses and schedules
- uploaded/seeded PDFs
- preview images if needed

### Why this split
- LangGraph is best used for **stateful, deterministic, multi-step document understanding**
- TinyFish is best used for **web-page interaction and browser-style reasoning over agency portals/pages**
- This keeps the demo coherent and avoids using TinyFish for work it is not meant to do

## Workflow Design

### 1. LangGraph PDF Processing Workflow
This graph starts when the user clicks `Process / Extract`.

Graph stages:
1. `load_document`
2. `ocr_document`
3. `extract_candidate_fields`
4. `normalize_fields`
5. `validate_mrn`
6. `validate_demographics`
7. `validate_contact_info`
8. `validate_services_and_physician`
9. `reconcile_conflicts`
10. `publish_extracted_referral`

Expected outputs:
- patient name
- MRN
- DOB
- phone
- address
- ZIP
- hospital/discharge source
- insurance name/plan
- PCP / ordering physician
- ordered services
- validation warnings
- field confidence / completeness

Important rule:
- LangGraph owns PDF understanding end to end
- final output is a normalized `ExtractedReferral` record persisted to Supabase

### 2. TinyFish Eligibility Workflow
This starts only after LangGraph finishes and the user has a structured patient record.

Inputs:
- extracted patient ZIP
- extracted insurance plan
- patient/referral structured summary

TinyFish pages to inspect:
- dummy `service area ZIP page`
- dummy `accepted insurance plans page`

Workflow:
1. open service ZIP page and verify patient ZIP
2. open accepted insurance page and verify insurance plan
3. combine both findings with extracted patient info
4. emit final eligibility decision:
   - `eligible`
   - `blocked_zip`
   - `blocked_insurance`
   - `manual_review`

Important rule:
- TinyFish is using the **nice, already-extracted structured patient info**
- it is not parsing the PDF

### 3. TinyFish Scheduling Workflow
This starts only if eligibility passes.

Inputs:
- structured patient info
- ordered services
- dummy nurse specialization page
- dummy nurse schedule page

Workflow:
1. read ordered services from extracted referral
2. determine required nurse capabilities
3. inspect nurse specialization page
4. inspect nurse schedule/availability page
5. choose the best matching nurse and slot
6. simulate patient outreach / call initiation
7. return final scheduling result

Outputs:
- assigned nurse
- rationale for match
- proposed call/visit slot
- patient outreach status
- scheduling initialized / blocked

- `/demo/coverage-zips`
  - searchable list/table of supported ZIP codes

- `/demo/nurse-specializations`

- `/demo/nurse-schedules`

These pages must be deterministic and easy for TinyFish to inspect.

## Test Plan

### Core flow
- homepage -> portal works
- new referral appears in portal
- referral detail shows PDF and current metadata
- processing starts LangGraph and persists extracted result
- result page renders extracted fields
- eligibility workflow correctly passes/fails based on ZIP + insurance
- scheduling workflow correctly matches ordered services to nurse specialization + availability
- final scheduled state displays correctly

### Failure states
- OCR/extraction partial failure still produces warnings and partial output
- missing phone/address/physician remains marked missing
- ZIP fail blocks scheduling
- insurance fail blocks scheduling
- no matching nurse creates blocked/manual-review scheduling state

### Refresh and persistence
- refreshing `processing`, `eligibility`, or `schedule` restores latest persisted state
- prior workflow events remain viewable from DB history

## Assumptions and defaults

- Frontend is migrated to Next.js App Router with TypeScript.
- UI uses shadcn/ui.
- LangGraph performs PDF OCR, extraction, and validation.
- TinyFish performs only downstream operational browser workflows.
- Supabase is used for Postgres and Storage.
- Backend remains separate from Next.js and runs on AWS App Runner.
- Dummy operational pages are part of the demo and intentionally stable.
- No user auth is required for the hackathon build.
- Seeded referral PDFs and outputs are used so the demo stays smooth and repeatable.
