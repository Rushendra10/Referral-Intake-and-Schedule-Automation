# Three-Person Plan with Agentic Work Split Evenly

## Summary

Keep the same product story and architecture, but split the agentic implementation across all three people instead of giving all orchestration work to one backend owner.

Core system behavior stays:

- **LangGraph** processes the PDF:
  - OCR
  - field extraction
  - validation
  - normalized referral output
- **TinyFish** does downstream browser-style operations:
  - ZIP coverage check
  - insurance acceptance check
  - nurse specialization and schedule lookup
  - patient outreach / scheduling simulation

Platform stays:

- **Frontend**: Next.js + shadcn/ui on Vercel
- **Backend**: Node service on AWS App Runner
- **Database / storage**: Supabase
- **Workflow runtime**: LangGraph
- **Browser automation**: TinyFish

The difference is ownership: each person gets one meaningful part of the agentic system, plus their adjacent UI/platform work.

## Ownership Boundaries

To avoid overlap, split the agentic system into three lanes:

- **Lane A: LangGraph document intelligence** -> Person 1
- **Lane B: TinyFish eligibility intelligence** -> Person 2
- **Lane C: TinyFish scheduling intelligence + persistence runtime** -> Person 3

Person 3 still owns the shared backend skeleton, but not all agent logic. Person 1 and Person 2 each own a real workflow, not just UI around someone else’s backend.

## App Flow

1. User lands on homepage and clicks `Demo`
2. Portal shows seeded referrals
3. User opens a referral and sees PDF + current fields
4. User clicks `Process / Extract`
5. **LangGraph** OCRs and extracts the PDF
6. Extracted patient/referral data is shown nicely on the result page
7. User starts eligibility
8. **TinyFish eligibility workflow** checks ZIP coverage and insurance acceptance using dummy pages
9. If eligible, user starts scheduling
10. **TinyFish scheduling workflow** checks nurse specialization and schedule pages, chooses a nurse/slot, and simulates outreach
11. Final screen shows scheduled / blocked result

### 2. TinyFish Eligibility Workflow — Person 2
Stages:
1. read extracted ZIP and insurance from structured referral
2. open ZIP coverage page
3. verify ZIP coverage
4. open accepted insurance page
5. verify plan acceptance
6. merge findings and emit eligibility result

Outputs:
- `eligible`
- `blocked_zip`
- `blocked_insurance`
- `manual_review`

Split note:
- Person 2 owns the TinyFish automation and target pages
- Person 3 exposes persistence and stream endpoints
- Person 1 integrates eligibility start/navigation from extracted result pages

### 3. TinyFish Scheduling Workflow — Person 3
Stages:
1. read extracted referral and ordered services
2. determine required nurse capability set
3. open nurse specialization page
4. open nurse schedules page
5. select best nurse and visit/call slot
6. simulate patient outreach / call initiation
7. emit final scheduling result

Outputs:
- assigned nurse
- rationale
- proposed slot
- outreach status
- scheduling initialized / blocked

Split note:
- Person 3 owns this workflow end to end
- Person 2 builds/skins the final scheduling UI
- Person 1 wires navigation and live page integration

## Test Plan

### Processing
- PDF processing produces extracted referral output
- partial OCR failure still yields warnings
- missing fields remain marked missing, not fabricated

### Eligibility
- ZIP accepted + insurance accepted -> eligible
- ZIP rejected -> blocked
- insurance rejected -> blocked
- ambiguous plan -> manual review

### Scheduling
- qualified nurse with availability -> assigned
- no qualified nurse -> blocked/manual review
- qualified nurse but no matching slot -> blocked/manual review

### Full flow
- homepage -> portal -> referral -> processing -> result -> eligibility -> schedule works
- refresh on any workflow page restores latest state
- prior workflow events are viewable from persisted history

## Assumptions

- Frontend migrates to Next.js App Router with TypeScript.
- UI uses shadcn/ui.
- LangGraph handles PDF understanding only.
- TinyFish handles downstream agency-operation checks only.
- Supabase is DB + storage.
- Backend remains a separate Node service on AWS App Runner.
- Dummy operational pages are intentionally stable and deterministic.
- No auth is required in hackathon scope.

## Default Coordination Rule

Each person must ship one real agentic subsystem, not just surrounding UI or plumbing:

- Person 1: LangGraph processing workflow
- Person 2: TinyFish eligibility workflow
- Person 3: TinyFish scheduling workflow + shared runtime

That keeps the agentic work distributed evenly while still leaving clean ownership boundaries.
